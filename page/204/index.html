Milan 击败 Liverpool
Table of contents

大概是我看过的最难看的一届冠军杯决赛了，比2003年的那一届0:0更加难看。倒不是因为场面沉闷，而是双方的失误多如牛毛。

感觉 Milan 是想要放缓并控制节奏，但是经常出现原地拿球发呆，然后被 Liverpool 球员从后面抢走的情况，Jankulovski 失误最多，多次滑倒、被断球；而 Liverpool 在进攻方面到了关键时候就人球分离，支离破碎，毫无气势可言。出现这种情况最有可能的就是体力不够，可是这两支队伍好像都花了不少时间调整状态和积蓄体力吧？

在场面被动的情况下，Inzaghi 的2个进球很关键。这位机灵的射手一直是欧洲赛场上最可怕的杀手之一，只要有他在场，对手就会担心随时遭到致命打击。今天的第1个进球是他用肩膀（从慢镜头回放来看其实是上臂啦）改变 Pirlo 任意球的方向射进的，虽然有运气成分在里面，但也是他积极抢点的回报；第2个进球之前，他浪费了几个好机会，不过一旦突破越位变成单刀，他难以控球摆脱后卫的弱点就不用计较了。

Milan 的其他队员都没有良好的表现。
Dida 能够扑出 Gerrard 的射门更多是因为后者的意图明显，发力又不足；
Benitez 对 Kaka 的“照顾”还是基本成功的，Alonso 对他的犯规和造越位的失误造成失球并不能够说明他的表现有多好，基本上是低于应有的水准的；
中场长时间失去控制力，Seedorf 的表现尤其差劲，进攻中基本消失之外，拿球也屡屡被抢下；
虽然最后时刻才让对手得分，但后卫其实是有很多失误的，没有被早早扳平只是 Liverpool 人的进攻做得太差了而已。

Ancelotti 的临场调度一如既往地看不出什么水平。状态调整不好本来就是失败，换人也没有解决问题的迹象。如果不是可能有进入加时赛的顾虑，前2次换人完全可以提前，如果真的被扳平，也许这就成了败笔。

红军方面，Benitez 少见的没有进行果断的调整。
上半场一结束，我就在群里面问，要攻坚的话，会上 Crouch 吗？
米兰上半场1:0，状态又不好，下半场肯定会坚决防守反击，既然能够压制住对方状态不好的中场，这个时候就应该加强边路冲击，换下一个状态最差的中场，上 Crouch 摆渡、会做给 Kuyt 支援，或分球给边路插上，这样才能够打开局面，同时在定位球方面多了一个抢点的，他自己又能够以匪夷所思的扭曲体位射门，一举多得。
可是这个换人直到80分钟左右才来到，此后虽然追回1球，但时间已经不够了。

站在全场的角度看，Liverpool 和对方一样缺少足够的决心，边路进攻不够坚决，甚至连球也不想要的感觉，Finnan 在前场一次配合中的发呆就是最佳例证，Kewell 上场之后基本消失，只有 Riise 的几次射门还有点威胁。
Pennant 和 Zenden 很卖力，但是他们不属于那种聪明的球员，多次机会都选择了最没有威胁的方案。
Gerrard 和 Alonso 等人在防守经验丰富的对手面前想要在中路做文章基本是失败的，而且 Gerrard 明显不在状态，抽射、头球、单刀都算是自己帮对手化解的。
于是大部分时间里，Liverpool 虽然占有场面上的优势，但是在这种不是他们擅长的节奏和局面下，始终找不到进球的办法。1:2算是一个公平结果。

在比分落后，局面无法打开，而且对手没有状态的情况下，Benitez 没有选择拼刺刀的方案，没有吹响反击的号角，丢掉了 Liverpool 最大的优势——气势，在一场连激烈都称不上（对比欧冠半决赛和足总杯决赛）的比赛中憋死，也就不奇怪了。

最后总结一下，说得重一点，两队都是一副缺少体力、缺少状态、缺少士气、缺少欲望、缺少决心的疲沓样子，感觉就是两个快要到达终点的长跑选手抱在一起滚过线。

Feed 乱码
Table of contents

因为近期出现 feed 在 Google Reader 变成问号的问题（我用 Reader 却一直没发现…所以说人品很重要的），所以怀疑是不是新增加的插件带来的问题。目前禁用了 Editing Manager 看看效果，无论有没有修正错误，希望能够得到你们的反馈，谢谢。

禁用插件时手一抖，误点中禁用 Permalinks Migration 的链接，赶紧重新激活。不知道会不会造成又一波的 trackback 更新狂潮。

最后，把 Subscribe To Comments 的默认选项改为不订阅，因为我不知道 trackback 大量更新会不会骚扰到订阅回复通知的读者，安全起见，今后还是默认关掉的好。

补充：表情功能好像失效了，正在查找原因、寻找替代的插件和调整页面，如果有可怕的事情发生，请勿惊恐。

对比 Diigo 和 Linebuzz
Table of contents

和菜头大力推荐 Linebuzz，因为这项服务允许网站访问者高亮页面上的一段话，并且针对这段话添加评论。

对，我的第一反应就是：这和第一狗的书签服务不是很相近么？
但是转念一想，也许 Linebuzz 的使用门槛更低，毕竟第一狗还需要安装工具栏或者书签。

不过在和菜头的 blog 页面上尝试一下才发现，其实 Linebuzz 的门槛也不见得比第一狗低多少。

首先，Linebuzz 的评论要求填写验证码，而第一狗不需要；

其次，Linebuzz 仍然需要先注册一下，这和我先前以为的无需注册/登录就可以直接使用的想法差远了，而既然都要注册的话，那么顺手增加一个第一狗脚本书签也不是多麻烦的事情。

另一方面，第一狗的功能则强过 Linebuzz 太多：

首先它是个以书签为主的服务，所有被高亮和评论过的页面都被当作在线书签处理，管理更加方便，还有网页快照、标签、群组、群组讨论、快速发布 blog、收藏视频、转发邮件、页面信息、导入导出、同时收藏到其他书签站点等众多功能；

其次，添加了 Linebuzz 脚本的页面载入速度非常慢，发布评论也要等很久，还常出现脚本失效需要重新载入的情况，这也和这个服务最近使用人数暴增有关系。第一狗的脚本书签也可能载入慢，但如果使用工具条的话，速度很快，可以显示该页面是否已经评论，而且就算第一次发布失败，还会不断尝试直到成功为止。

第三，Linebuzz 的页面效果是不可控的，如果评论过多，或者其他人的高亮遮挡了你的，你不喜欢它的高亮形式，载入速度太慢，影响选择文字，作为浏览者是很头痛的。而第一狗可以选择让其他人或者所有的高亮、评论隐身。

第四，不是所有的页面都可以增加脚本代码，虽然这样的做法方便了浏览者，但也同时制约了 Linebuzz 的适用范围。而第一狗的工具栏和书签已经有多种主流浏览器的版本，包括 Firefox、Opera、IE、Safari、Flock。

第五，通过第一狗发布的评论可以通过 Linkrolls 和 Daily Blog Post 自动展示、发布到自己的 blog 上面。

第六，要查看页面上的评论，还需要多点击一次，“Click for more…”，而第一狗的评论是直接展现的，没有多余操作。

第七，Linebuzz 提交评论之后需要重新刷新页面，而第一狗不重新载入就可以完成收藏、高亮、评论操作，无疑更加节约时间和流量，特别是图片和视频很多的页面。

第八，目前还没有找到 Linebuzz 提交评论之后进行修改、进行私有评论的功能，而第一狗一开始就具备此功能。

第九，Linebuzz 的收藏没有 RSS Feed 输出功能，而向主流书签站点看齐的第一狗在这方面做得很周到。

第十，第一狗的开发人员是华人，如 Maggie，在它的论坛中进行交流时比 Linebuzz 更加方便。

Webware 100的投票活动将在明天开始，我肯定会给第一狗投上一票。

另：还记得那个Web 2.0 … The Machine is Us/ing Us 的视频么？其中第一狗也作为代表性的 Web 2.0服务露了一小脸。
